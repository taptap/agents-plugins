# 变更分析模板集

本文件定义 `change-analysis` 工作流（Story 场景）输出的两份人类可读 markdown 报告格式：

1. `code_change_analysis.md` — 飞书云文档「代码变更分析报告」
2. `test_coverage_report.md` — 飞书云文档「测试覆盖评估报告」

> 第三份「补充测试用例」由后端从 `supplementary_cases.json` 渲染，模板不在本 skill 控制；本 skill 只保证 JSON 字段正确（详见 [CONVENTIONS.md 用例 JSON 格式](../../CONVENTIONS.md#用例-json-格式)）。

> **关于符号约定**：本 skill 输出的所有 markdown 报告状态标记使用中文方括号（`[实证]/[推测]`、`[高]/[中]/[低]`、`[已覆盖]/[未覆盖]/[待评估]`），**禁止使用 emoji ✅/⚠️/❌、ASCII `[OK]/[!]/[X]`、星级 ★/★★/★★★**——前两类飞书 import 会破坏渲染（详见 §「关键约束」），后一类与跨 workflow 分级符号不统一。

### 各章节格式规则

**第 0 章「变更评估」**

- 第一行格式固定：`变更风险：{标记} {判定}`，标记三选一：`[低]` 可常规回归 / `[中]` 重点回归 N 项 / `[高]` 须主测 + 灰度
- 紧跟一段自然语言概括 + 实证 vs 推测占比（X% / Y%）
- 元数据末尾必须包含 `分析方式：AI 辅助 + Dev/QA 评审决策`

**第 1 章「变更分组总览」**

- 按业务域字母编号分组（A/B/C/...），每组用 `### {字母}. {组名}` H3 标题
- 组内用 bullet 列出文件 + 一句话变更性质 + 行数变化（如 `+235/-72`）
- **不使用 markdown 表格**

**第 2 章「核心变更深度分析」**

- 每个核心文件用 `### 2.X {文件名}（[高]/[中]/[低]）` H3 + 风险标记
- 文件下用 `**变更点 N：{简述}**` 粗体标题（不用 H4，避免大纲膨胀）
- 每条结论行尾标 `[实证]` / `[推测]`
- **代码 diff 块约定**：`> 以下「旧逻辑/新逻辑」代码块仅 Dev 排查问题时阅读，PM/QA 可跳过本子节。` 加在代码块前作引用提示
- API 模型类轻量变更用列表，不用表格

**第 3 章「调用链分析」**

- 用 markdown 代码块（` ```text `）保留 ASCII 树结构
- 调用链中含 [推测] 节点的明确标注

**第 4 章「影响面评估」**

- §4.1 直接影响 / §4.2 间接影响 / §4.3 高风险场景 / §4.4 回归范围
- §4.1 / §4.2 用 bullet 列表（不用表格），格式 `- **{功能域}**：{描述} — [高/中/低] [实证/推测]`
- §4.3 每条独占一行 + [实证/推测] 标记
- §4.4 必测 / 建议测分级，每条 `→ 详见 [测试覆盖评估] 缺口 N` 跨文档引用

**第 5 章「置信度索引」**

- 仅作索引，不重复结论详情（结论已在 §1-§4 各章节行尾标注）
- 给出 [实证] vs [推测] 的章节列表 + 占比
- 末尾用 quote block 提示评审会重点确认 [推测] 部分

## 关键约束

- 整份报告使用纯文本和分隔线排版，**不使用 markdown 表格**（飞书 import 后表格被拆成离散 text block，看起来像散文）
- 不写 H1 标题（飞书 doc 名已包含报告类型）
- **状态符号统一使用中文方括号标记**（非 emoji 非 ASCII 非星级）：
  - 风险等级：`[高]` / `[中]` / `[低]`（替代旧的 ★★★/★★/★ 与 `[OK]/[!]/[X]`）
  - 实证置信度：`[实证]` 基于代码 diff 可见 / `[推测]` 仅基于推断未实证
  - 覆盖状态：`[已覆盖]` / `[未覆盖]` / `[待评估]`
  - 变更评估：`[低]` 可常规回归 / `[中]` 重点回归 / `[高]` 须主测 + 灰度
  - 覆盖率结论：`[已达标]` / `[有缺口]` / `[严重不足]`
- **禁止使用 emoji** ✅/⚠️/❌：实测飞书 import 会破坏内容
- **禁止使用 ASCII** `[OK]/[!]/[X]`：跨 workflow 语义漂移（rr 表"有条件"、ca 表"低风险但难复现"）
- **禁止使用星级** ★/★★/★★★：与跨 workflow 分级符号不统一
- **代码 diff 块前必须加引用提示** `> 以下代码块仅 Dev 排查问题时阅读，PM/QA 可跳过`
- **multi-doc 必须双向引用**：
  - code_change_analysis.md §4.4 回归范围 → 引用 test_coverage_report.md §2 缺口编号
  - test_coverage_report.md §2 缺口分析 → 引用 code_change_analysis.md §2/§4 变更点
  - test_coverage_report.md §4 补充建议 → 引用 ca_sup（supplementary_cases.json 渲染版）的 TC-XX 编号
- **置信度逐条标注**：[实证] 必须有 diff 行号或代码注释为依据；[推测] 必须说明无法实证的原因
- **覆盖率统计必须精确**：禁止"约 N 个"模糊描述；总变更点必须在 §0 元数据声明并枚举来源
