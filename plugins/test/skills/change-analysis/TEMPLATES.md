# 变更分析模板集

本文件定义 `change-analysis` 工作流（Story 场景）输出的两份人类可读 markdown 报告格式：

1. `code_change_analysis.md` — 飞书云文档「代码变更分析报告」
2. `test_coverage_report.md` — 飞书云文档「测试覆盖评估报告」

> 第三份「补充测试用例」由后端从 `change_supplementary_cases.json` 渲染，模板不在本 skill 控制；本 skill 只保证 JSON 字段正确（详见 [CONVENTIONS.md 用例 JSON 格式](../commons/CONVENTIONS.md#用例-json-格式)）。

> **关于符号约定**：本 skill 输出的所有 markdown 报告状态标记使用中文方括号（`[实证]/[推测]`、`[高]/[中]/[低]`、`[已覆盖]/[未覆盖]/[待评估]`），**禁止使用 emoji ✅/⚠️/❌、ASCII `[OK]/[!]/[X]`、星级 ★/★★/★★★**——前两类飞书 import 会破坏渲染（详见 §「关键约束」），后一类与跨 workflow 分级符号不统一。

### 各章节格式规则

**第 0 章「变更评估」**

- 第一行格式固定：`变更风险：{标记} {判定}`，标记三选一：`[低]` 可常规回归 / `[中]` 重点回归 N 项 / `[高]` 须主测 + 灰度
- 紧跟一段自然语言概括 + 实证 vs 推测占比（X% / Y%）
- 元数据末尾必须包含 `分析方式：AI 辅助 + Dev/QA 评审决策`

**第 1 章「变更分组总览」**

- 按业务域字母编号分组（A/B/C/...），每组用 `### {字母}. {组名}` H3 标题
- 组内用 bullet 列出文件 + 一句话变更性质 + 行数变化（如 `+235/-72`）
- 本章是树状层级（业务域→文件），保留 bullet 不用表格

**第 2 章「核心变更深度分析」**

- 每个核心文件用 `### 2.X {文件名}（[高]/[中]/[低]）` H3 + 风险标记
- 文件下用 `**变更点 N：{简述}**` 粗体标题（不用 H4，避免大纲膨胀）
- 每条结论行尾标 `[实证]` / `[推测]`
- **代码 diff 块约定**：`> 以下「旧逻辑/新逻辑」代码块仅 Dev 排查问题时阅读，PM/QA 可跳过本子节。` 加在代码块前作引用提示
- API 模型类轻量变更用 4 列表格（模型名 / 字段变更 / 影响 / 实证）

**第 3 章「调用链分析」**

- 用 markdown 代码块（` ```text `）保留 ASCII 树结构
- 调用链中含 [推测] 节点的明确标注

**第 4 章「影响面评估」**

- §4.1 直接影响 / §4.2 间接影响 / §4.3 高风险场景 / §4.4 回归范围
- §4.1 / §4.2 用 4 列表格：`功能域 / 影响描述 / 风险 / 实证`
- §4.3 每条独占一行 + [实证/推测] 标记（场景描述较长，bullet 更合适）
- §4.4 必测 / 建议测分级，每条 `→ 详见 [测试覆盖评估] 缺口 N` 跨文档引用

**第 5 章「置信度索引」**

- 仅作索引，不重复结论详情（结论已在 §1-§4 各章节行尾标注）
- 给出 [实证] vs [推测] 的章节列表 + 占比
- 末尾用 quote block 提示评审会重点确认 [推测] 部分

**第 6 章「交叉验证摘要」（条件触发）**

- 仅当 Codex 子 Agent 执行成功（`codex_change_findings.json` 存在）时追加；未启用 / 未跑通 → 整节省略
- §6.1 双方共同确认 / §6.2 仅 Codex / §6.3 仅 Claude 用 3 列表格汇总
- **§6.4 综合结论**：报告的核心行动清单，每条 finding 嵌套二选一 checkbox（评审会勾选定性）：
  - `- [ ] 有效问题，处理方案：____________`
  - `- [ ] 无效问题，原因：____________`
- §6.4 之后加一行最终风险等级（`最终风险等级维持 [高/中/低]; 建议...`），与 §0 评估一致
- 6.4 是该报告的"待评审会拍板的发现清单"——前面 §4.x 是分析过程，6.4 是结论

## 关键约束

通用渲染规范见 [CONVENTIONS.md「飞书文档渲染规范」](../commons/CONVENTIONS.md#飞书文档渲染规范)（表格 vs bullet 选择规则、禁用元素清单）。本 skill 特定补充：

- 第 1 章「变更分组总览」是树状层级（业务域→文件），保留 bullet
- 第 2.5 节 / 第 4.1 / 4.2 / 第 3 章「覆盖率汇总」用表格（详见上方各节格式规则）
- 第 4.3 「高风险场景识别」描述列较长，保留 bullet 一行一条
- **状态符号统一使用中文方括号标记**（具体取值见各章节）：
  - 风险等级：`[高]` / `[中]` / `[低]`（替代旧的 ★★★/★★/★ 与 `[OK]/[!]/[X]`）
  - 实证置信度：`[实证]` 基于代码 diff 可见 / `[推测]` 仅基于推断未实证
  - 覆盖状态：`[已覆盖]` / `[未覆盖]` / `[待评估]`
  - 变更评估：`[低]` 可常规回归 / `[中]` 重点回归 / `[高]` 须主测 + 灰度
  - 覆盖率结论：`[已达标]` / `[有缺口]` / `[严重不足]`
- **禁止使用 ASCII** `[OK]/[!]/[X]`：跨 workflow 语义漂移（rr 表"有条件"、ca 表"低风险但难复现"）
- **禁止使用星级** ★/★★/★★★：与跨 workflow 分级符号不统一
