# 测试失败分析各阶段详细操作指南

## 关于系统预取

通用预取机制见 [CONVENTIONS.md](../../CONVENTIONS.md#系统预取)。本 skill 不依赖 Story 预取，输入来源为测试执行结果和代码 diff。

## 阶段 1: init - 初始化

### 1.0 输入路由

按 [CONVENTIONS.md](../../CONVENTIONS.md#本地文件输入) 定义的优先级确认输入来源：

1. `test_output` 参数提供了测试执行的 stdout/stderr 文本 → 直接使用
2. `test_report` 参数提供了测试报告文件（JSON/XML/文本）→ Read 该文件
3. 以上均不满足 → 停止并报错：「测试执行结果为空，无法分析」

**测试命令发现**（供 5.2 自循环重测使用）：

1. `test_command` 参数已提供 → 记录备用
2. `test_report` 文件中包含 `command` 字段 → 提取记录
3. 以上均无 → 在 5.2 阶段提示用户手动执行测试并提供新的输出

### 1.1 确认代码变更来源

按以下优先级获取 diff：

1. `code_diff` 参数提供了 diff 文件 → Read 该文件
2. `code_diff_text` 参数提供了 diff 文本 → 直接使用
3. 以上均无 → 尝试从 git 获取：`git diff HEAD~1` 或 `git diff --cached`
4. 仍无 diff → 降级运行：跳过 diff 关联分析，所有失败归类为「待确认」

### 1.2 确认迭代轮次

1. 检查工作目录是否存在 `previous_analysis` 参数指定的文件或 `failure_analysis.json`
2. 如存在 → Read 该文件，提取 `iteration` 字段，本轮为 `iteration + 1`
3. 如不存在 → 本轮为第 1 轮
4. 检查 `max_iterations` 参数（默认 3，由 SKILL.md 自循环协议定义；用户可在触发时通过自然语言指定，如「最多修复 2 轮」），如当前轮次 > max_iterations → 停止并输出最终报告

### 1.3 确认上游需求数据（可选）

检查工作目录是否存在 `requirement_points.json`（上游 requirement-clarification 产出）：
- 存在 → Read 作为分类时的辅助参考（用于判断失败是否与需求变更相关）
- 不存在 → 跳过，不影响核心分类流程

## 阶段 2: collect - 数据收集

本阶段目标：从测试执行结果中提取结构化的失败用例信息，从代码 diff 中提取变更范围。

### 2.1 解析测试执行输出

根据测试框架类型解析输出，提取每个失败用例的以下字段：

| 字段 | 说明 | 必填 |
| --- | --- | --- |
| `test_name` | 测试方法/函数名 | 是 |
| `test_file` | 测试文件路径 | 是 |
| `error_message` | 失败的错误消息 | 是 |
| `stack_trace` | 堆栈信息 | 否（有则提取） |
| `assertion_detail` | 具体断言：expected vs actual | 否（有则提取） |

支持的测试框架输出格式：

| 框架 | 输出特征 |
| --- | --- |
| Go (`go test`) | `--- FAIL:` 行 + 缩进的错误详情 |
| Python (pytest) | `FAILED` 标记 + `E` 前缀错误行 |
| JavaScript (jest/vitest) | `● ` 前缀 + `Expected:` / `Received:` |
| Swift (XCTest) | `XCTAssertEqual failed:` 等 |
| Java (JUnit) | `Tests run:` 统计行 + 异常堆栈 |
| 通用 JSON/XML 报告 | 按结构化字段直接解析 |

如格式无法自动识别 → 按纯文本逐行扫描 `FAIL`、`ERROR`、`failed`、`assertion` 等关键词提取。

### 2.2 解析代码 diff

从 diff 中提取变更范围信息：

1. **变更文件列表**：每个文件的路径
2. **变更函数/方法列表**：从 hunk header（`@@` 行）中提取函数名
3. **变更行号范围**：每个文件的修改行范围
4. **变更类型**：新增 / 删除 / 修改

### 2.3 如为第 N+1 轮迭代

回读上一轮的 `failure_analysis.json`：
- 提取上一轮的分类结果和失败数
- 对比本轮失败列表，识别哪些是新增失败、哪些是持续失败、哪些已修复

### 2.4 输出

将提取的失败用例信息写入 `raw_failures.md`，包含：

```markdown
# 原始失败用例（第 N 轮）

## 测试执行概要
- 总测试数：X
- 通过：X
- 失败：X
- 跳过：X

## 代码变更概要
- 变更文件数：X
- 变更函数/方法：[列表]

## 失败用例列表

### 1. {test_name} ({test_file})
- **错误消息**：{error_message}
- **断言详情**：expected={X}, actual={Y}
- **堆栈**：
  ```
  {stack_trace}
  ```

### 2. ...
```

## 阶段 3: classify - 失败分类

本阶段目标：对每个失败用例进行分类并打置信度评分。

### 3.1 逐用例分类

对 `raw_failures.md` 中的每个失败用例，执行以下判定流程：

**步骤 1：diff 关联检查**

- 从失败的堆栈/错误消息中提取涉及的源码文件和行号
- 检查这些文件和行号是否在 diff 的变更范围内
- 关联方式：
  - **直接关联**：失败断言指向的代码行在 diff 中被修改
  - **间接关联**（> 1 跳）：失败断言指向的函数调用了 diff 中修改的函数（confidence 上限 75）
  - **无关联**：失败路径与 diff 无交集

**步骤 2：分类判定**

```
if 直接关联 or 间接关联:
    → 候选分类：预期变化
    → 进一步检查：失败的预期值是否因需求变更而需要更新
    → 如有 requirement_points.json，交叉验证变更是否对应某个需求点
elif 匹配不稳定模式:
    → 候选分类：不稳定
    → 检查：超时 / 网络 / 时序 / 随机 / 历史波动
else:
    → 候选分类：回归问题
    → 需要根因分析：代码变更如何间接影响了该测试
```

**步骤 3：置信度评分**

| 证据级别 | 分数区间 | 示例 |
| --- | --- | --- |
| diff 行号直接命中失败断言 | 90-100 | 修改了函数返回值，测试断言该返回值 |
| diff 修改的函数被失败路径调用 | 70-89 | 修改了工具函数，调用方测试失败 |
| 基于错误模式匹配 | 50-69 | 错误消息包含 timeout，推测为不稳定 |
| 无明确证据 | <50 | 丢弃，标记为 [待确认] |

### 3.2 不稳定测试特殊处理

当候选分类为「不稳定」时，额外检查：

1. 如为第 2+ 轮迭代，检查该测试在上一轮是否通过 → 通过则加强不稳定判定（confidence += 15）
2. 如错误消息匹配多个不稳定模式 → confidence += 10
3. 如同一测试文件中有多个测试失败且模式相似 → 可能是环境问题而非单个测试不稳定

### 3.3 回归问题根因分析

当候选分类为「回归问题」时，使用 Opus 模型深入分析：

1. **追溯调用链**：从失败测试的代码路径回溯，寻找与 diff 的间接关联
2. **评估影响范围**：修改的代码被多少其他模块依赖
3. **判定严重程度**：

| 严重程度 | 判定标准 | 处理方式 |
| --- | --- | --- |
| high | 核心业务流程受影响，或影响多个模块 | 仅输出修复建议，不自动修复 |
| medium | 非核心功能受影响，影响范围有限 | 输出修复建议，用户确认后可执行 |
| low | 边缘场景，对业务影响极小 | 可建议暂时跳过，后续处理 |

### 3.4 降级回退

- diff 不可用 → 跳过关联分析，所有失败标记为 [待确认]，confidence 上限 60
- 堆栈信息缺失 → 仅基于错误消息和测试文件名分析，confidence 上限 70

## 阶段 4: plan - 方案生成

本阶段目标：基于分类结果生成 `failure_analysis.json` 和 `action_plan.md`。

### 4.1 生成 failure_analysis.json

按 SKILL.md 定义的格式组装分类结果：

1. 填充 `iteration`（当前轮次）
2. 填充 `total_failures`（本轮总失败数）
3. 按 `expected_change`、`regression`、`flaky` 三个分类填充详情
4. 计算 `summary`：
   - 各分类计数
   - `convergence`：如为第 2+ 轮，比较本轮失败数与上轮 → 减少则为 `true`，否则为 `false`

### 4.2 生成 action_plan.md

按 SKILL.md 定义的格式生成行动方案：

1. **预期变化区**：列出需更新的测试及建议的新预期值
2. **回归问题区**：按严重程度降序排列，包含根因分析和修复建议
3. **不稳定区**：标注是否建议重试，以及长期修复建议
4. 末尾附「下一步」确认清单

### 4.3 呈现结果

将 `action_plan.md` 呈现给用户，等待确认：

- 用户确认全部 → 进入阶段 5（loop）
- 用户部分确认 → 仅对确认项执行修复
- 用户拒绝 → 终止，输出当前 `failure_analysis.json` 作为最终结果

## 阶段 5: loop - 自循环编排

本阶段目标：执行用户确认的修复操作，重测并检查收敛。

### 5.1 执行修复

按用户确认的行动方案执行：

**预期变化修复**：
1. 定位测试文件中的失败断言
2. 按 `suggested_fix` 更新预期值
3. 修改前再次确认修改内容

**回归问题修复**（仅 medium/low severity 且用户确认）：
1. 按 `suggested_fix` 修改源码
2. 修改前再次确认修改内容
3. high severity 的回归问题仅记录，不自动修复

**不稳定测试处理**：
1. 标记为需重试
2. 不修改测试代码（除非用户明确要求）

### 5.2 重新执行测试

修复完成后，重新执行测试：
- 如知道测试执行命令 → 直接执行
- 如不知道 → 提示用户执行测试并提供新的测试输出

### 5.3 收敛检查

获取新的测试结果后：

1. 统计新的失败数
2. **集合比较**：提取本轮和上轮的 `test_name` 集合
   - 计算新增失败集合：`new_failures = current_set - previous_set`
   - 计算已修复集合：`fixed = previous_set - current_set`
3. 与上一轮比较：
   - 失败数 = 0 → **全部通过**，输出最终 `failure_analysis.json` 并退出
   - 失败数 < 上轮 且 `new_failures` 为空 → **收敛中**，继续下一轮（回到阶段 2）
   - 失败数 < 上轮 但 `new_failures` 非空 → **部分收敛 + 新增回归**，在报告中标注新增回归项，继续下一轮但降低修复信心
   - 失败数 ≥ 上轮 → **不收敛**，停止循环，输出最终报告
4. 检查迭代次数：
   - 已达 `max_iterations` → 停止循环，输出最终报告

### 5.4 最终报告

循环终止时，输出最终版 `failure_analysis.json` 和 `action_plan.md`，包含：

- 所有轮次的分析汇总
- 已修复的项目清单
- 仍未解决的项目清单及建议
- 终止原因（全部通过 / 达到最大次数 / 不收敛）
