# 用例评审模板集

本文件定义 `test-case-review` 工作流 `summary` 阶段产出的 `review_summary.md`（最终上传到飞书云文档的人类可读报告）格式。

> **关于符号约定**：本 skill 输出报告（review_summary.md）的状态标记使用中文方括号 `[已覆盖]/[部分覆盖]/[未覆盖]`、`[通过]/[有条件]/[不通过]`，**禁止使用 emoji ✅/⚠️/❌**——飞书 import 会把 ✅→`[OK]`、⚠️→`[!]`、❌触发 bitable block 解析导致整行内容丢失（详见 §「关键约束」）。

---

## review_summary.md 模板

> `summary` 阶段输出。这是最终上传到飞书云文档的报告，必须严格按本模板组织。

### 完整报告示例骨架

> 设计原则：飞书云文档名已经是「{需求名称}-用例评审-{YYYYMMDDHH}」，故 review_summary.md **不写 H1 标题**，避免飞书导入后出现重复标题。

```markdown
## 0. 评审结论

用例就绪度：[有条件] 有条件通过
存在 N 个 P0 阻塞项需立即修复，M 个 P1 改进项建议本轮跟进。核心问题集中在……（一句话概括问题域）。

待修复（按优先级）：
- P0 阻塞 N 个：{一句话总结，如「3 个用例端到端不闭环 + 1 个核心需求未覆盖」}
- P1 改进 M 个：{一句话总结}
- P2 优化 K 个：{一句话总结}

**元数据**
- 需求名称：XXX
- 需求链接：https://...
- 评审日期：YYYY-MM-DD
- 评审范围：{用例总数} 条 × {评审维度个数} 维度
- 分析方式：AI 辅助 + 评审人决策

## 1. 评审总览

**核心指标**
- 需求功能点：N 个（RP-1 ~ RP-N）
- 测试用例：M 条（P0: a / P1: b / P2: c / P3: d）
- 需求覆盖率：x/N 完整覆盖（X%），y/N 部分覆盖，z/N 未覆盖（**目标 ≥ 90%**）
- 场景缺失：K 个（P1: a / P2: b / P3: c）
- 正确性问题：J 处
- 规范性问题：L 处

## 2. 需求覆盖率

> **用例引用规则**：所有 `{用例ID}` 必须用 markdown link `[{用例ID}]({MS编辑链接})` 呈现，URL 用 prefetch 阶段注入的"用例编辑链接模板"拼出（`https://{ms_base_url}/#/track/case/edit/{caseId}?projectId={projectId}`）。如果 ms_cases_snapshot.json 里 case 已有 `edit_url` 字段，**直接用该字段**，不要自己拼。无 prefetch / local source 时降级为纯 ID 文本。

### 2.1 已覆盖（x/N）

| 需求点 | 描述 | 关联用例 |
|---|---|---|
| RP-1 | {需求点描述} | [{用例ID-1}]({MS编辑链接}), [{用例ID-2}]({MS编辑链接}) |
| RP-2 | {需求点描述} | [{用例ID}]({MS编辑链接}) |

### 2.2 部分覆盖（y/N）

| 需求点 | 描述 | 关联用例 | 缺什么 / 建议补充 |
|---|---|---|---|
| RP-K | {需求点描述} | [{用例ID}]({MS编辑链接}) | {具体说明 + 建议补充什么} |

### 2.3 未覆盖（z/N）

| 需求点 | 描述 | 建议新增用例方向 |
|---|---|---|
| RP-M | {需求点描述} | {用例方向} |

## 3. 场景完整性

### 3.1 端到端链路不闭环
- [{用例ID}]({MS编辑链接})（{用例名}）：只有 N 个步骤，缺少 ①{X}；②{Y}；③{Z}
- ...

### 3.2 缺失边界场景

| 关联需求 | 优先级 | 缺失场景描述 |
|---|---|---|
| RP-7 | [P1] | 白名单模式只选启动器包未选游戏本体包时尝试提交的系统行为 |
| RP-7 | [P2] | 更新日志超过 5000 字符截断/提示 |
| RP-10 | [P2] | 白名单→非白名单切换时若有审核中包体的处理 |

## 4. 用例正确性

### 4.1 预期结果有误/不完整
- [{用例ID}]({MS编辑链接})：当前预期"{原预期}"——{问题}。修改建议：{建议}
- ...

### 4.2 步骤术语错误
- [{用例ID}]({MS编辑链接})（{用例名}）步骤N：{问题描述} + 修改建议
- ...

### 4.3 优先级合理性
- [{用例ID}]({MS编辑链接})（{用例名}）当前 P3，建议调整为 P2。理由：{业务含义}
- ...

## 5. 用例规范性

### 5.1 命名问题
- [{用例ID}]({MS编辑链接})（当前名："{原名}"）：{问题} → 建议改为"{新名}"

### 5.2 步骤规范问题
- [{用例ID}]({MS编辑链接})：{问题描述} → 建议拆分为步骤①{X}；②{Y}

### 5.3 前置条件规范问题
- [{用例ID}]({MS编辑链接})：{问题描述} → 建议补充"{补充内容}"

## 6. 修复 to-do（按用例 ID 分组）

> 同一个用例可能在 §3 / §4 / §5 多处被点名，本节按用例 ID 一次列全所有需要修的问题，并在评审会上勾选决策。**用例 ID 必须用 markdown link 跳转 MS**（同 §2-§5 规则）。

**P0 阻塞**

- **用例 [{用例ID}]({MS编辑链接})（{用例名}）**
  - **修步骤**：{具体改成什么，例如"拆为 5 步：① 进入提交流程 ② 等待审核中态触发 ③ 退出后再回查"}
  - **修预期**：{具体改成什么，例如"补'显示版本号 v2.3.4 + 审核中标识'"}
  - [ ] 已修复
  - [ ] 无需修复，原因：____________

- **用例 [{用例ID2}]({MS编辑链接})（{用例名}）**
  - **补步骤**：{具体改成什么，例如"端到端补 ④ 退出后回到列表 ⑤ 验证条目状态"}
  - [ ] 已修复
  - [ ] 无需修复，原因：____________

**P1 改进**

- **用例 [{用例ID3}]({MS编辑链接})（{用例名}）**
  - **改术语**：步骤 N 中"游戏整包" → "游戏包"
  - [ ] 已修复
  - [ ] 无需修复，原因：____________

**P2 优化**

- **用例 [{用例ID4}]({MS编辑链接})（{用例名}）**
  - **改名称**：含问号"是否能重提？" → 陈述句"包体审核失败后重提流程"
  - [ ] 已修复
  - [ ] 无需修复，原因：____________

## 7. 建议补充的用例（本次跳过创建）

> 如已开启自动生成，对应用例已写入 `review_supplementary_cases.json`，本节仅做清单展示。新生成的用例尚未入 MS，无 link，保留纯文本方向描述。

**P1**
- {用例方向} — 关联需求点 RP-N

**P2**
- {用例方向} — 关联需求点 RP-N
```

### 各章节格式规则

**第 0 章「评审结论」**

- 第一行格式固定：`用例就绪度：{标记} {判定}`，标记三选一：`[通过]` / `[有条件]` / `[不通过]`
  - **禁止使用 emoji** ✅/⚠️/❌ 与 ASCII `[OK]/[!]/[X]`：实测飞书 import 转换问题
- 紧跟一段自然语言总结（1-2 句），概括核心问题域
- 待修复列表按 P0/P1/P2 分级，每级一句话总结，**不展开问题详情**（详情在 §6）
- 元数据末尾必须包含 `分析方式：AI 辅助 + 评审人决策`，不写「评审人：QA AI 助手」之类的署名

**第 1 章「评审总览」**

- KV 计数指标，用 bullet 列表（不适合表格化）
- 必须包含覆盖率 + 目标值（默认 ≥ 90%；项目可在 PHASES.md 调整目标线）
- 用例数按优先级 P0/P1/P2/P3 分布
- 问题数按 4 个维度分别给出（覆盖/完整性/正确性/规范性）

**第 2 章「需求覆盖率」**

按"已覆盖 / 部分覆盖 / 未覆盖"三组分别用表格呈现：
- §2.1 已覆盖：3 列表格 `需求点 / 描述 / 关联用例`
- §2.2 部分覆盖：4 列表格 `需求点 / 描述 / 关联用例 / 缺什么 / 建议补充`
- §2.3 未覆盖：3 列表格 `需求点 / 描述 / 建议新增用例方向`

**第 3 章「场景完整性」**

- §3.1 端到端不闭环：列具体用例 ID，描述缺什么步骤（描述长，bullet）
- §3.2 缺失边界场景：3 列表格 `关联需求 / 优先级 / 缺失场景描述`

**第 4 章「用例正确性」**

按 4.1 / 4.2 / 4.3 三个子维度组织，每条问题独占一行 `- {用例ID}：{描述} → 修改建议`。

**第 5 章「用例规范性」**

按 5.1 命名 / 5.2 步骤 / 5.3 前置条件 三个子维度组织。

**第 6 章「修复 to-do（按用例 ID 分组）」**

- 报告的核心可执行价值章节，**评审会上勾选决策**
- 同一个用例的所有问题（跨 §3 / §4 / §5）汇总到一起
- 按 P0 / P1 / P2 分级
- 用例行用 **粗体 + markdown link + 用例名**：`- **用例 [{用例ID}]({MS编辑链接})（{用例名}）**`
- 用例行下**先列具体待办**，每条格式 `- **{动作动词}**：{具体改成什么，含目标态/新文本}`（让修复者一行就知道要改什么）
  - 动作动词：`修步骤` / `补步骤` / `修预期` / `改术语` / `改名称` / `补前置` / `调优先级` 等具体动词
  - **禁止只写维度标签**（如 `[正确性] §4.1.1`）—— 维度名 + 章节引用是 AI 内部组织视角，对修复者无价值；修复者要的是"改什么、改成什么"
  - 如改动较复杂，可在末尾加"详情 §X.X.X"作为回溯锚点（可选）
- 待办行之后嵌套 2 个状态 checkbox（追踪修复进度，二选一）：
  - `- [ ] 已修复`
  - `- [ ] 无需修复，原因：____________`
- 选项语义：评审会读完后修复者主动勾"已修复"；如认为不需要修，必须填原因（issue 编号、技术债理由、需求变更等）
- §6 与 §3/§4/§5 内容会有重叠（按用例视角 vs 按维度视角），这是有意的冗余——修复者只看 §6 就能拿走全部待办 + 标进度

**第 7 章「建议补充的用例」**

- 与 §2.3 未覆盖呼应，但更具体（用例方向描述）
- 按 P1 / P2 分组（P0 通常已在 review_supplementary_cases.json 自动生成）
- 标注关联的 RP-N

### 关键约束

通用渲染规范见 [CONVENTIONS.md「飞书文档渲染规范」](../commons/CONVENTIONS.md#飞书文档渲染规范)（表格 vs bullet 选择规则、禁用元素清单）。本 skill 特定补充：

- §1 评审总览 → 保留 bullet（KV 指标）
- §2 需求覆盖率（3 个子表）+ §3.2 缺失边界场景 → 用表格（详见上方各章节格式规则）
- §3.1 端到端不闭环 / §4 用例正确性 / §5 用例规范性 / §6 修复 to-do → 保留 bullet（描述长 / 嵌套结构 / 索引语义）
- 不写 H1 标题（飞书 doc 名已包含「{需求名称}-用例评审-{YYYYMMDDHH}」）
- **状态符号统一使用中文方括号标记**（具体取值见各章节）：
  - 用例就绪度：`[通过]` / `[有条件]` / `[不通过]`
  - 覆盖状态：`[已覆盖]` / `[部分覆盖]` / `[未覆盖]`
  - 优先级：`[P0]` / `[P1]` / `[P2]` / `[P3]`
- **禁止 §6 复述 §3/§4/§5 的问题详情** —— §6 是按用例 ID 重新组织的索引，子条目只引用章节编号 + 一句话动作，避免大段重复
- **不写「评审人：QA AI 助手」**之类的 AI 署名 —— 用「分析方式：AI 辅助 + 评审人决策」强调结论由人在评审会最终决策
- 用例 ID 在报告全文中保持一致（hex hash 或其他业务 ID），跨章节可通过 ID 交叉引用
